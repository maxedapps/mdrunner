# Releasing mdr

This is the maintainer source of truth for cargo-dist setup, qualification, and versioned GitHub Releases. Ordinary development still uses Cargo; cargo-dist owns packaging and release publication only.

## Ownership and release contract

- `dist-workspace.toml` is repository-owned configuration. Edit it, then run `dist init --yes`.
- `.github/workflows/release.yml` is generated by cargo-dist 0.32.0. Never hand-edit it; verify it with `dist generate --check`.
- `.github/workflows/ci.yml` is repository-owned and uses one Ubuntu runner for formatting, clippy, and locked tests. It intentionally performs no release-mode or cross-platform binary builds.
- `Cargo.toml`, the root `mdr` entry in `Cargo.lock`, `mdr --version`, the changelog release heading, and the proposed tag must resolve to one immutable semantic version.
- `publish = false` remains intentional. `[package.metadata.dist] dist = true` and `packages = ["mdr"]` opt the binary into GitHub distribution without enabling crates.io publication.
- Release tags are v-prefixed annotated tags and must point at the reviewed release commit.
- The matching `CHANGELOG.md` version section is the authoritative release prose. Generated commit lists may supplement it but never replace it.
- The repository owner must explicitly approve the exact commit, release notes, target-status table, and unsigned-artifact caveats before any tag is created or pushed.

## Prepare the release candidate

1. Update `Cargo.toml` and only the root `mdr` package version in `Cargo.lock`.
2. Move release entries from `Unreleased` into `## [X.Y.Z] - YYYY-MM-DD` in `CHANGELOG.md`. Leave a fresh empty `Unreleased` section and advance its comparison link to `vX.Y.Z...HEAD`.
3. Confirm the release section is concise, user-visible, and includes the target-status and unsigned-artifact caveats. Stable cargo-dist release prose must come from the matching version section.
4. Run the local gate:

   ```bash
   cargo metadata --locked --no-deps --format-version 1
   cargo fmt --check
   cargo clippy --all-targets -- -D warnings
   cargo test --locked
   cargo build --release --locked
   cargo run --quiet -- --version
   dist --version
   dist init --yes
   dist generate --check
   dist plan --tag=vX.Y.Z
   dist build --tag=vX.Y.Z --print=linkage
   git diff --check
   ```

   `dist --version` must report 0.32.0. A local `dist build` is current-host evidence only; it cannot prove all configured targets.
5. Inspect the current-host archive, metadata files, executable name, sidecar digest, `mdr --help`, and `mdr --version`. Confirm native linkage for the selected clipboard and Rustls HTTP dependencies. `target/distrib/` is generated residue and must remain untracked.
6. Require `.github/workflows/ci.yml` and the final plan-mode release workflow to pass on the exact candidate commit. Ordinary CI proves the Rust test suite on Ubuntu; plan mode validates cargo-dist configuration without starting binary-builder jobs. The version-tag workflow is the only hosted path that performs native release builds and packaging.

## One-time setup pull-request proof

Use this only for initial pipeline setup or a major release-pipeline change:

1. Require ordinary Ubuntu CI to pass or run in parallel.
2. Temporarily set `pr-run-mode = "upload"` in `dist-workspace.toml` and run `dist init --yes`.
3. Push the setup commit to the pull request. Confirm the release workflow runs plan plus all local/global artifact jobs while `host` and `announce` remain skipped. A pull-request upload must not create a tag or GitHub Release.
4. Download all `artifacts-*` workflow artifacts. cargo-dist 0.32.0 uses granular uploads; do not assume the UI provides one literal `artifacts.zip`.
5. Record run/job URLs, runner/target mapping, archive and installer names, file sizes, linkage output, skips, and every checksum result. Inspect each archive for the expected executable plus `README.md`, `CHANGELOG.md`, and `LICENSE`, and confirm there are no unrelated binaries or secrets.
6. Return `pr-run-mode` to `"plan"`, rerun `dist init --yes` and `dist generate --check`, push the restoration, and require final ordinary CI plus release planning to pass before merge.

## Build-tested versus native-qualified

A target is **build-tested** only after cargo-dist build/package/checksum evidence succeeds for that target. Ordinary Ubuntu CI does not establish target availability. A target is **native-qualified** only after the release candidate records each applicable mode separately on matching desktop hardware:

- run outside-repository `.md` and inert `.mdx` files with representative local assets;
- run representative Markdown through stdin;
- preserve safely restorable clipboard text, then exercise clipboard Markdown, one textual path, one `file://` URL, one native `.md`, one native `.mdx`, an unsupported native file, and multiple native files; restore and verify the saved state afterward;
- fetch one bounded local HTTP fixture, one public HTTPS Markdown document, and the canonical GitHub blob example; inspect remote-relative links/images, credentials/redaction, timeout, and oversize failures;
- inspect deterministic generated HTML, static CSP, local embedding, remote URL resolution, and absence of source credentials or executable MDX/HTML;
- exercise `--no-open` with default output and verify successful generation/path printing without a browser invocation;
- exercise relative and absolute `--out` destinations, missing parent creation, a path containing spaces, and atomic replacement of an existing custom file; verify no deterministic cache copy is written, then combine custom output with `--no-open`;
- observe the actual default browser open the encoded `file://` URL for both default and custom output, browser retrieval of an allowed remote image, prompt process exit, and absence of a product server;
- inspect the installed/current-host archive's help/version and representative successful source paths.

If the prior clipboard cannot be safely captured and restored, obtain approval before overwriting it; otherwise mark native clipboard cases unqualified. Record unavailable source modes as `build-tested, unqualified`; never infer clipboard, URL, custom-output replacement, or browser qualification from fake tests, compilation, another OS, Rosetta, or packaging. X11/XWayland and supported Wayland data-control sessions may qualify independently; unsupported pure-Wayland/headless sessions should produce the documented clipboard error. Preserve the historical v0.1.0 evidence below. Record the Linux glibc baseline from hosted linkage output; do not claim broad distro compatibility from the target triple.

## Approval and publication

Before publication, assemble the reviewed commit hash, changelog prose, planned artifact inventory, ordinary-CI and plan-run URLs, available current-host/setup evidence, qualification table, Linux baseline, and unsigned-artifact caveats. Obtain explicit owner approval. The tag-triggered workflow supplies the authoritative current-release native build, package, checksum, and attestation evidence; inspect it before treating every target as build-tested. If desired, the owner may enable immutable GitHub Releases only after confirming the current cargo-dist draft/upload/release sequence is compatible.

Only after approval, tag the reviewed commit:

```bash
git tag -a v0.3.0 -m "Release 0.3.0"
git push origin v0.3.0
```

The generated tag workflow creates the GitHub Release; do not create a competing release manually.

## Post-publication verification

1. Confirm `main`, `v0.3.0`, and the approved commit are the intended objects.
2. Confirm the release title/body uses the matching changelog section and lists the approved target statuses and unsigned-artifact caveats.
3. Download every asset and verify each archive sidecar plus `sha256.sum` using an OS-native SHA-256 tool.
4. Inspect all four archives and run `mdr --version` on available native systems.
5. Test the published shell and PowerShell installer commands where native access exists.
6. Verify hosted provenance, for example:

   ```bash
   gh attestation verify <downloaded-asset> --repo maxedapps/mdr
   ```

7. Confirm the new empty `Unreleased` comparison link starts at `v0.3.0`.

Checksums detect corruption but do not authenticate the publisher. GitHub attestations provide hosted build provenance but are not macOS notarization, Windows Authenticode, or another platform signature.

## Recovery

### Before publication

Stop, fix the candidate, synchronize version/changelog/config again, rerun affected local and hosted checks, and obtain renewed owner approval. A local unpublished tag may be removed only if it was never pushed or used for a public release. If release preparation is postponed, move unfinished notes back to `Unreleased` rather than leaving a misleading dated section.

### After publication starts

Never move, replace, or reuse a published tag/version. Preserve the failure evidence. Resume only a safe, idempotent job for the same immutable commit when the owner approves it; otherwise publish a corrected patch version with new notes and artifacts. An incomplete draft may be deleted only after confirming no release became public or immutable. Security-sensitive assets may be withdrawn, but withdrawal does not make the original version reusable.
